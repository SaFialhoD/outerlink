## Goal

Define the documentation standards for OutterLink: API documentation (Rust doc comments), user-facing guides, developer/contributor documentation, and the rules governing how docs are written, maintained, and reviewed. - Developer documentation covers: architecture, contributing, adding CUDA intercepts, adding transport backends
- Documentation rules are codified and enforced in PR review

## 1. API Documentation (Rust Doc Comments)

### Doc Comment Strategy

Every public item (`pub fn`, `pub struct`, `pub enum`, `pub trait`, `pub mod`) MUST have a `///` doc comment. No exceptions.

### Rules for Doc Comments

```rust
/// Brief one-line summary of what this does.
///
/// Longer explanation if needed. Explain WHY this exists, not just WHAT it does.
/// The reader already sees the function signature -- tell them things they
/// cannot infer from the types alone.
///
/// # Arguments
///
/// * `device_index` - Zero-based CUDA device index (maps to `CUdevice`).
///   Must be less than the value returned by `cuDeviceGetCount`.
///
/// # Returns
///
/// The device name as reported by the CUDA driver, or a `CudaError` if
/// the device index is out of range.
///
/// # Errors
///
/// Returns `CudaError::InvalidDevice` if `device_index` >= device count.
/// Returns `CudaError::NotInitialized` if `cuInit` has not been called.
///
/// # Examples
///
/// ```no_run
/// use outterlink_common::cuda::get_device_name;
///
/// let name = get_device_name(0).expect("Failed to get device name");
/// assert!(name.contains("NVIDIA"));
/// ```
///
/// # Panics
///
/// Does not panic. All error conditions are returned as `Result`.
pub fn get_device_name(device_index: i32) -> Result<String, CudaError> {
    // ...
}
```

### Which Sections Are Required

| Section | When Required |
|---------|--------------|
| Brief summary (first line) | Always |
| Extended description | When behavior is non-obvious |
| `# Arguments` | When function has >0 parameters |
| `# Returns` | When return type is not `()` |
| `# Errors` | When function returns `Result` |
| `# Examples` | All public functions (use `no_run` if needs GPU) |
| `# Panics` | When function can panic, or explicitly state it does not |
| `# Safety` | All `unsafe fn` -- explain invariants the caller must uphold |

### Module-Level Documentation

Each `lib.rs` and `mod.rs` must have `//!` module-level docs:

```rust
//! # OutterLink Common
//!
//! Shared types, protocols, and transport abstractions used by both
//! the OutterLink client (interception library) and server (GPU daemon).
//!
//! ## Crate Structure
//!
//! - [`transport`] - Pluggable transport backends (TCP, io_uring, UCX)
//! - [`protocol`] - Wire protocol message types and serialization
//! - [`cuda_ffi`] - CUDA Driver API FFI bindings and stubs
//! - [`pool`] - GPU pool management and scheduling
//!
//! ## Feature Flags
//!
//! - `cuda` - Enable real CUDA FFI bindings (requires CUDA toolkit)
//! - `ucx` - Enable UCX transport backend (requires libucx)
```

### `cargo doc` Configuration

```toml
# Cargo.toml (workspace root)
[workspace.metadata.docs.rs]
all-features = true
rustdoc-args = ["--cfg", "docsrs"]
```

```rust
// lib.rs - enable doc_cfg for feature-gated items
#![cfg_attr(docsrs, feature(doc_cfg))]

/// UCX transport backend.
///
/// Requires the `ucx` feature flag and libucx installed on the system.
#[cfg(feature = "ucx")]
#[cfg_attr(docsrs, doc(cfg(feature = "ucx")))]
pub mod ucx_transport;
```

### Enforcing Documentation in CI

```toml
# .cargo/config.toml
[build]
rustdocflags = ["-D", "missing_docs"]
```

Or in CI:

```yaml
- run: RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
```

This fails the build if any public item lacks a doc comment.

### Hosting

API docs are generated by `cargo doc` and deployed to GitHub Pages by the `docs.yml` workflow defined in P3. The landing page redirects to `outterlink_common/index.html`.

### 01-installation.md

| Section | Content |
|---------|---------|
| Prerequisites | Linux (Ubuntu 22.04+), NVIDIA GPU with driver 550+, CUDA 12.x |
| Hardware requirements | Minimum: 1 GPU per node. Recommended: ConnectX-5 for RDMA |

Organized as problem/solution pairs:

**Solution:**

2. Verify the server is running: `outterlink-cli status`
3. Check network: `ping <server-ip>`
4. Check firewall: `sudo ufw status`
```

### How to Add a New CUDA Function Intercept

**File:** `docs/guides/dev-add-cuda-function.md`

Step-by-step guide for adding interception of a new CUDA Driver API function:

```markdown
# How to Add a New CUDA Function Intercept

## Overview

OutterLink intercepts CUDA Driver API functions by providing alternative
implementations that serialize the call, send it to the server, and
deserialize the result. This guide walks through adding a new function.

## Steps

### 1. Identify the Function

Find the function signature in the CUDA Driver API documentation or
in `cuda.h`. Example: `cuDeviceGetUuid`.

```c
CUresult cuDeviceGetUuid(CUuuid* uuid, CUdevice dev);
```

### 2. Add to the Dispatch Table

In `crates/outterlink-client/csrc/cuda_dispatch.c`, add the function
to the dispatch table:

```c
INTERCEPT(cuDeviceGetUuid, cuDeviceGetUuid_t)
```

### 3. Add the Stub Header

In `csrc/stub-headers/cuda.h`, add the function pointer type:

```c
typedef CUresult (*cuDeviceGetUuid_t)(CUuuid*, CUdevice);
```

### 4. Define the Protocol Message

In `crates/outterlink-common/src/protocol/messages.rs`:

```rust
#[derive(Serialize, Deserialize)]
pub struct DeviceGetUuidRequest {
    pub device: i32,
}

#[derive(Serialize, Deserialize)]
pub struct DeviceGetUuidResponse {
    pub result: i32,
    pub uuid: [u8; 16],
}
```

### 5. Implement the Client-Side Handler

In `crates/outterlink-client/src/intercept/device.rs`:

```rust
pub fn handle_cuDeviceGetUuid(uuid: *mut CUuuid, dev: CUdevice) -> CUresult {
    let request = DeviceGetUuidRequest { device: dev as i32 };
    let response: DeviceGetUuidResponse = transport.call(&request)?;
    unsafe { *uuid = std::mem::transmute(response.uuid); }
    response.result
}
```

### 6. Implement the Server-Side Handler

In `crates/outterlink-server/src/handlers/device.rs`:

```rust
pub fn handle_device_get_uuid(req: DeviceGetUuidRequest) -> DeviceGetUuidResponse {
    let mut uuid: CUuuid = unsafe { std::mem::zeroed() };
    let result = unsafe { cuDeviceGetUuid(&mut uuid, req.device) };
    DeviceGetUuidResponse {
        result,
        uuid: unsafe { std::mem::transmute(uuid) },
    }
}
```

### 7. Add Tests

Add a unit test in `crates/outterlink-common/src/protocol/tests.rs`
for serialization/deserialization.

Add an integration test (gated behind `#[cfg(feature = "cuda")]`)
that calls the function end-to-end.

### 8. Update the Function Count

Update the interception function count in the README and in the
`SUPPORTED_FUNCTIONS.md` list.

## Checklist

- [ ] Function added to C dispatch table
- [ ] Stub header updated
- [ ] Protocol messages defined
- [ ] Client handler implemented
- [ ] Server handler implemented
- [ ] Serialization test added
- [ ] Integration test added (if hardware available)
- [ ] Documentation updated
```

### How to Add a New Transport Backend

**File:** `docs/guides/dev-add-transport.md`

```markdown
# How to Add a New Transport Backend

## Overview

OutterLink uses a pluggable transport architecture. All transports
implement the `Transport` trait defined in `outterlink-common`.

## The Transport Trait

```rust
// crates/outterlink-common/src/transport/mod.rs

#[async_trait]
pub trait Transport: Send + Sync {
    /// Connect to a remote endpoint
    async fn connect(addr: &str) -> Result<Self, TransportError>
    where Self: Sized;

    /// Send a buffer to the remote endpoint
    async fn send(&self, buf: &[u8]) -> Result<usize, TransportError>;

    /// Receive data from the remote endpoint
    async fn recv(&self, buf: &mut [u8]) -> Result<usize, TransportError>;

    /// Send a buffer and wait for a response (RPC pattern)
    async fn send_and_recv(&self, buf: &[u8]) -> Result<Vec<u8>, TransportError>;

    /// Name of this transport (for logging)
    fn name(&self) -> &str;
}
```

## Steps

### 1. Create the Transport Module

Create `crates/outterlink-common/src/transport/my_transport.rs`.

### 2. Implement the Trait

Implement `Transport` for your new transport struct.

### 3. Add Feature Flag

In `Cargo.toml`, add a feature flag for any external dependencies:

```toml
[features]
my_transport = ["dep:my-transport-sys"]
```

### 4. Register in Transport Selection

In `crates/outterlink-common/src/transport/mod.rs`, add your
transport to the `create_transport` function:

```rust
#[cfg(feature = "my_transport")]
if config.transport == "my_transport" {
    return Box::new(MyTransport::connect(&config.address).await?);
}
```

### 5. Add Tests

Write tests that verify your transport passes the same test suite
as the TCP transport.

### 6. Document

Add a section to `docs/guides/06-performance-tuning.md` explaining
when to use your transport and how to configure it.
```

## 4. Documentation Rules

### Rules from `.claude/rules/documentation-framework.md`

The existing documentation framework (referenced below) defines the overall structure. These rules remain in effect:

- Every document has: header, purpose, content, related docs, open questions
- Folder structure: `planning/`, `docs/`, `side-docs/`
- Writing rules: no orphan knowledge, update don't duplicate, date everything, link aggressively, tables over paragraphs

### Additional Rules for Code Documentation

#### Rule 1: No Public API Without Docs

Every `pub` item must have a `///` doc comment. Enforced by `#![warn(missing_docs)]` in each crate's `lib.rs` and `-D missing_docs` in CI.

#### Rule 2: Examples Must Compile

Doc examples must compile (tested by `cargo test --doc`). Use `no_run` for examples that need GPU hardware. Use `ignore` only as a last resort with a comment explaining why.

```rust
/// # Examples
///
/// ```no_run
/// // This example needs a running OutterLink server
/// use outterlink_client::connect;
/// let client = connect("192.168.100.2:9700").unwrap();
/// ```
```

#### Rule 3: Unsafe Code Requires Safety Comments

Every `unsafe` block must have a `// SAFETY:` comment explaining why the unsafe operation is sound:

```rust
// SAFETY: cuMemAllocHost guarantees the returned pointer is page-aligned
// and valid for `size` bytes. We own this allocation until cuda_host_free.
unsafe {
    cuMemAllocHost(&mut ptr, size);
}
```

Every `unsafe fn` must have a `# Safety` doc section.

#### Rule 4: Error Types Are Documented

Every variant of every error enum must have a doc comment explaining when it occurs and what the caller should do:

```rust
pub enum TransportError {
    /// The remote server closed the connection unexpectedly.
    /// Retry the connection or check if the server is running.
    ConnectionClosed,

    /// The network operation timed out after the configured duration.
    /// This may indicate network congestion or a server crash.
    Timeout(Duration),
}
```

#### Rule 5: Changelog for User-Visible Changes

Maintain a `CHANGELOG.md` using the Keep a Changelog format. Every PR that changes user-visible behavior must add an entry under `[Unreleased]`.

```markdown
# Changelog

## [Unreleased]

### Added
- UCX transport backend with automatic RDMA negotiation

### Fixed
- cuMemcpyDtoH returning stale data when batch flush races with sync
```

#### Rule 6: Doc Comments Describe Behavior, Not Implementation

Good: "Allocates device memory on the remote GPU and returns a handle."
Bad: "Calls serialize_memalloc, sends via transport.send_and_recv, deserializes response."

The implementation can change; the behavior should not.

#### Rule 7: Diagrams Use ASCII Art in Docs

For architecture diagrams in Markdown files, use ASCII art so they render in any viewer (terminal, GitHub, editor). Mermaid diagrams are acceptable in GitHub-rendered docs as a secondary option.

```
Client App
    |
    | cuMemcpyHtoD(dst, src, size)
    v
Interception Layer (.so)
    |
    | serialize + batch
    v
Transport (TCP / io_uring / UCX)
    |
    | network
    v
Server Daemon
    |
    | cuMemcpyHtoD(real_dst, data, size)
    v
GPU (real CUDA)
```

## 5. Documentation Review in PRs

### PR Checklist for Documentation

Add to `.github/pull_request_template.md`:

```markdown
## Documentation Checklist

- [ ] New public APIs have `///` doc comments with `# Examples`
- [ ] `unsafe` blocks have `// SAFETY:` comments
- [ ] Error variants are documented
- [ ] CHANGELOG.md updated (if user-visible change)
- [ ] User guides updated (if behavior change)
- [ ] `cargo doc --no-deps` produces no warnings
```

### Automated Checks in CI

From P3's `ci.yml`:

```yaml
- name: Check documentation
  run: RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features --document-private-items

- name: Test doc examples
  run: cargo test --doc
```

## Test Plan

| Test | Expected |
|------|----------|
| `cargo doc --no-deps -D warnings` | No warnings |
| `cargo test --doc` | All doc examples compile and pass |
| PR without doc comments on public API | Clippy or doc warnings fail CI |
| PR with `unsafe` block without SAFETY comment | Review catches it (not automated, but in checklist) |
| GitHub Pages deployment | API docs accessible at `https://<owner>.github.io/outterlink/` |

## Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Docs drift from implementation | Users confused | `cargo test --doc` catches example drift; PR checklist |
| Over-documentation slows development | Velocity drop | Only PUBLIC API requires full docs; internal code needs only brief comments |
| Doc examples bit-rot | False confidence | Doc tests run in CI on every PR |
